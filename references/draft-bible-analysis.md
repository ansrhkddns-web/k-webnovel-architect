# 원고와 바이블 분석

이미 작성된 회차나 설정집을 분석할 때 사용한다. 목적은 특정 작품의 데이터를 스킬에 저장하는 것이 아니라, 현재 프로젝트가 어떤 방식으로 설계되었고 무엇이 부족한지 판단하는 것이다.

## 분석 원칙

- 고유명사, 구체 수치, 회차 사건, 문장을 스킬 산출물에 복사하지 않는다.
- 분석 결과는 해당 사용자 프로젝트에만 말한다.
- 스킬 문서에 반영할 때는 `일반화된 작업 방식`, `템플릿`, `진단 기준`으로만 바꾼다.
- 원고 평가는 문장 취향보다 구조, 보상, 훅, 연속성, 이탈 위험을 먼저 본다.

## 회차별 추출

각 회차에서 아래 정보를 추출한다.

```markdown
| 화 | 회차 기능 | 시작 압박 | 보상/변화 | 체감 증거 | 주변 반응 | 엔딩 훅 | 이탈 위험 |
|---|---|---|---|---|---|---|---|
| 1 |  |  |  |  |  |  |  |
```

추출 기준:
- 회차 기능: 발견, 첫 사용, 첫 보상, 첫 적대, 중간 사이다, 확장, 회수 등.
- 시작 압박: 주인공을 움직이게 하는 문제.
- 보상/변화: 끝났을 때 달라진 것.
- 체감 증거: 독자가 변화를 실제로 느끼는 장면.
- 주변 반응: 누가 주인공 변화를 알아차리는가.
- 엔딩 훅: 다음 화에서 확인하고 싶은 구체적 질문.
- 이탈 위험: 고구마, 반복, 설명 과밀, 보상 지연 등.

## 설정집 엔진 분석

설정 문서는 아래 엔진으로 분류한다.

| 엔진 | 질문 |
|---|---|
| 독자 계약 엔진 | 이 작품이 어떤 독자에게 어떤 재미를 약속하는가? |
| 세계관 엔진 | 어떤 규칙이 반복 사건을 만드는가? |
| 성장 엔진 | 무엇이 오르고, 어떻게 체감되고, 어떤 대가가 생기는가? |
| 갈등 엔진 | 적과 장애물이 어떤 방식으로 커지는가? |
| 인물 엔진 | 각 인물이 어떤 기능을 맡는가? |
| 회차 엔진 | 각 화가 어떤 변화와 훅을 주는가? |
| 떡밥 엔진 | 무엇을 심고 언제 회수하는가? |
| 연속성 엔진 | 변경과 충돌을 어떻게 관리하는가? |

## 초반부 구조 진단

초반 15화 안팎은 보통 아래 기능을 증명해야 한다. 장르에 따라 화수는 달라질 수 있다.

| 구간 | 핵심 기능 | 부족할 때 보정 |
|---|---|---|
| 진입부 | 주인공 처지, 핵심 소재, 첫 위기 | 핵심 사건을 앞당긴다 |
| 첫 생존/첫 선택 | 주인공이 관찰자가 아니라 행동자임을 증명 | 작은 선택과 비용을 넣는다 |
| 첫 보상 | 독자가 이 작품의 보상 구조를 이해 | 숫자/정보 뒤 체감 장면을 붙인다 |
| 실험/활용 | 보상이 반복 재미가 될 수 있음을 증명 | 일상 문제나 관계 문제에 적용한다 |
| 외부 반응 | 변화가 혼자만의 정보가 아님을 증명 | 관찰자, 의심, 질투, 인정 반응을 넣는다 |
| 첫 아크 압박 | 보상 때문에 더 큰 문제가 생김 | 후폭풍과 새 적대자를 배치한다 |

## 주의할 판단

- 긴 회차가 항상 좋은 것은 아니다. 변화와 보상이 없으면 긴 회차도 늘어진다.
- 짧은 회차가 항상 나쁜 것은 아니다. 강한 훅과 명확한 변화가 있으면 기능할 수 있다.
- 설정집이 많아도 원고에서 체감되지 않으면 독자 보상이 아니다.
- 떡밥이 많아도 초반 보상이 부족하면 독자는 기다리지 않는다.
- 인물이 많아도 기능이 다르면 좋고, 기능이 겹치면 피로가 된다.
