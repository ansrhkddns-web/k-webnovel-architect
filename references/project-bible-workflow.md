# 프로젝트 바이블 워크플로

이 문서는 한국 웹소설 기획을 `작품 바이블` 형태로 쌓는 절차를 다룬다. 특정 작품의 설정을 저장하는 문서가 아니라, 어떤 작품에도 적용할 수 있는 작업 구조다.

## 원칙

- 바이블은 설정 백과가 아니라 연재를 계속하기 위한 작업 도구다.
- 먼저 독자 계약을 정하고, 그다음 세계관과 인물을 붙인다.
- 폴더나 문서명은 고정 규칙이 아니라 권장 구조다. 사용자 프로젝트 상황에 맞게 줄이거나 합친다.
- 참고 작품이나 기존 프로젝트를 분석할 때는 고유명사, 수치, 사건, 문장을 가져오지 않는다. 폴더 구조, 판단 순서, 체크 방식만 일반화한다.

## 권장 폴더 구조

```text
작품-바이블/
├── 00_overview/
├── 01_world/
├── 02_characters/
├── 03_power-system/
├── 04_arcs/
├── 05_episodes/
├── 06_continuity/
└── 07_drafts/
```

## 폴더별 역할

| 폴더 | 역할 | 핵심 질문 |
|---|---|---|
| `00_overview` | 작품 정체성, 제목, 콘셉트, 독자 계약 | 이 작품은 어떤 독자에게 어떤 재미를 주는가? |
| `01_world` | 세계관 규칙, 자원, 조직, 역사, 금기 | 어떤 규칙이 반복 사건을 만드는가? |
| `02_characters` | 주인공, 조력자, 라이벌, 적대자, 관계망 | 각 인물은 어떤 서사 기능을 맡는가? |
| `03_power-system` | 능력, 성장, 랭크, 돈, 명성, 관계 수치 | 무엇이 올라가고, 어떻게 체감되는가? |
| `04_arcs` | 시즌, 장기 로드맵, 클라이맥스, 엔딩 | 어떤 국면 전환이 독자를 오래 붙잡는가? |
| `05_episodes` | 초반 회차표, 문체 기준, 회차 목표 | 매화 어떤 변화와 훅을 줄 것인가? |
| `06_continuity` | 변경 로그, 떡밥, 설정 충돌 점검 | 나중에 앞뒤가 안 맞는 일을 어떻게 막을 것인가? |
| `07_drafts` | 실제 원고 또는 회차 초안 | 설계가 장면으로 구현되었는가? |

## 최소 문서 세트

새 작품을 빠르게 시작할 때는 아래만 먼저 만든다.

```text
00_overview/concept.md
00_overview/trope-pack.md
01_world/core-rules.md
02_characters/main-cast.md
03_power-system/growth-loop.md
04_arcs/series-roadmap.md
05_episodes/opening-roadmap.md
06_continuity/foreshadowing.md
06_continuity/consistency-checklist.md
```

## 작업 순서

1. `concept.md`에서 장르, 핵심 독자층, 한 줄 콘셉트, 제목 후보를 정한다.
2. `trope-pack.md`에서 익숙한 재미, 차별화, 위험, 보정책을 정한다.
3. `core-rules.md`에서 세계가 반복적으로 사건을 만드는 규칙을 정한다.
4. `main-cast.md`에서 인물을 성격보다 서사 기능 중심으로 나눈다.
5. `growth-loop.md`에서 보상이 어떻게 발생하고 체감되는지 정한다.
6. `series-roadmap.md`에서 시즌별 질문, 사이다, 다음 위기 씨앗을 정한다.
7. `opening-roadmap.md`에서 초반 회차별 변화와 엔딩 훅을 정한다.
8. 원고를 쓰거나 수정할 때마다 `foreshadowing.md`와 `consistency-checklist.md`를 갱신한다.
9. 상업성을 점검할 때는 `commercial-market-fit.md`, `paid-conversion-model.md`, `quality-gates-and-stress-tests.md`를 함께 본다.
10. 외부 레퍼런스를 참고할 때는 `reference-research-protocol.md`로 고유 데이터를 제거하고 일반화한다.

## 작업 레이어

바이블은 한 번에 완성하지 않는다. 아래 순서로 얇게 만든 뒤 필요할 때 깊게 만든다.

| 레이어 | 목표 | 산출물 |
|---|---|---|
| 1차 방향 | 무엇을 팔 작품인지 정한다 | 장르, 독자층, 한 줄 콘셉트, 제목 후보 |
| 2차 엔진 | 어떻게 50화 이상 굴러가는지 정한다 | 세계관 규칙, 성장 루프, 갈등 엔진 |
| 3차 인물 | 누가 어떤 기능을 맡는지 정한다 | 주인공, 적대자 사다리, 조력자, 감정축 |
| 4차 회차 | 초반부가 실제로 읽히게 만든다 | 1화 목표, 2-5화, 6-10화, 11-25화 |
| 5차 연속성 | 장기 연재에서 앞뒤를 맞춘다 | 떡밥표, 변경 로그, 설정 충돌 체크 |
| 6차 패키징 | 독자가 클릭하고 결제할 약속을 정한다 | 소개글, 태그, 유료화 후보 |
| 7차 검증 | 실제 연재에서 버틸지 점검한다 | 10초/5화/25화/50화 테스트 |

## 기존 프로젝트 분석 순서

기존 설정집이나 원고를 볼 때는 아래 순서로 분석한다.

1. 폴더와 문서가 어떤 역할로 나뉘었는지 본다.
2. 개요 문서에서 장르 계약과 핵심 보상을 찾는다.
3. 성장/능력 문서에서 반복 보상 루프를 찾는다.
4. 인물 문서에서 기능이 겹치는 인물과 비어 있는 기능을 찾는다.
5. 회차표와 원고를 비교해 실제 구현 여부를 본다.
6. 연속성 문서에서 떡밥, 변경 로그, 충돌 체크 체계를 본다.
7. 고유 데이터를 제외하고 일반화 가능한 작업 방식만 정리한다.

## 인물 기능 카드

인물은 외모와 성격보다 `회차에서 하는 일`이 먼저다.

```markdown
## [인물명 또는 역할명]
- 서사 기능:
- 주인공에게 주는 압박:
- 주인공의 성장을 확인시키는 방식:
- 독자 보상:
- 장기 기능:
- 퇴장 또는 전환 조건:
```

## 시즌 카드

```markdown
## [시즌명]

- 주 압박:

- 큰 사이다:

- 이탈 위험:
```

## 바이블 점검

- 콘셉트 문서와 회차표가 같은 장르 약속을 말하는가?
- 세계관 규칙이 설명에 그치지 않고 회차 사건을 만드는가?
- 인물마다 서사 기능이 겹치지 않는가?
- 성장 보상이 실제 장면에서 체감되는가?
- 장기 떡밥이 너무 많아 초반 보상을 밀어내지 않는가?
- 변경한 설정이 기존 회차와 충돌하지 않는가?

## 고도화 우선순위

바이블이 부족해 보일 때는 아래 순서로 보강한다.

1. 독자 계약: 누가 왜 읽는지 불분명하면 모든 문서가 흔들린다.
2. 반복 보상: 매화 무엇을 줄지 없으면 회차가 늘어진다.
3. 주인공 엔진: 욕망과 대가가 약하면 사건이 외부에서만 온다.
4. 적대자 사다리: 적이 단조로우면 사이다도 약해진다.
5. 초반 회차표: 1-10화가 약하면 장기 설정은 독자에게 닿지 않는다.
6. 떡밥/연속성: 장기 연재로 갈수록 신뢰를 지킨다.
7. 패키징: 제목과 소개글이 본문 약속과 맞아야 클릭이 생긴다.
8. 품질 게이트: 10초, 5화, 25화, 유료화 신뢰 테스트를 통과시킨다.
