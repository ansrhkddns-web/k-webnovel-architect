# 연속성 관리 워크플로

연속성 관리는 설정을 많이 쓰기 위한 장식이 아니라, 장기 연재에서 앞뒤가 어긋나 독자가 이탈하는 일을 막는 장치다.

## 기본 파일

```text

├── consistency-checklist.md
├── foreshadowing.md
├── names-and-terms.md
├── power-and-status-ledger.md

```

필요한 파일만 만든다. 짧은 작품이면 `changelog.md`, `consistency-checklist.md`, `foreshadowing.md` 세 개만으로 충분하다.

## 변경 로그

설정, 이름, 능력, 장기 전개, 회차 목표가 바뀌면 짧게 남긴다.

```markdown
## YYYY-MM-DD

- 변경:
- 이유:
- 영향 받는 문서:
- 이미 쓴 회차 수정 필요 여부:
```

## 설정 충돌 체크리스트

회차 작성 후 아래를 확인한다.

- 이번 화의 장르 보상이 기존 독자 계약과 맞는가?
- 주인공이 아직 알 수 없는 정보를 서술자가 확정하지 않았는가?
- 능력, 랭크, 돈, 평판, 관계 수치가 이전 회차와 이어지는가?
- 인물의 행동이 기존 욕망과 두려움에서 벗어나지 않았는가?
- 떡밥을 심었다면 기록했는가?
- 떡밥을 회수했다면 기존 표의 상태를 바꿨는가?
- 새 고유명사, 조직, 아이템, 기술명이 기존 이름과 충돌하지 않는가?
- 큰 사이다 뒤에 다음 위기 씨앗이 남았는가?

## 떡밥 관리

떡밥은 `숨기기`가 아니라 `나중에 독자가 납득할 수 있게 약속을 남기는 일`이다.

```markdown
| ID | 종류 | 심은 위치 | 겉뜻 | 숨은 진실 | 알림 주기 | 회수 목표 | 상태 |
|---|---|---|---|---|---|---|---|
| F-001 |  |  |  |  |  |  | 예정 |
```

상태:
- `예정`: 아직 원고에 넣지 않았다.
- `진행`: 원고에 심었고 아직 회수하지 않았다.
- `부분회수`: 일부 답을 줬지만 더 큰 질문이 남았다.
- `회수`: 독자가 납득할 만큼 처리했다.
- `폐기`: 더 쓰지 않기로 했으므로 다른 단서와 충돌하지 않게 정리해야 한다.

## 이름과 용어 관리

```markdown
| 분류 | 이름 | 첫 등장 | 기능 | 주의 |
|---|---|---|---|---|
| 인물 |  |  |  |  |
| 조직 |  |  |  |  |
| 능력 |  |  |  |  |
| 장소 |  |  |  |  |
```

이름표는 독자를 위한 설정집이 아니라 작가가 같은 기능의 인물을 중복 생성하지 않게 하는 도구다.

## 능력/상태 기록

성장물이면 능력 상태를 따로 기록한다.

```markdown
| 회차 | 변화 | 원인 | 체감 장면 | 주변 반응 | 다음 위험 |
|---|---|---|---|---|---|
|  |  |  |  |  |  |
```

능력 기록에서 가장 중요한 칸은 `체감 장면`이다. 숫자나 등급이 올랐는데 장면 변화가 없으면 독자는 보상을 느끼기 어렵다.

## 관계 기록

관계가 중요한 작품이면 감정선을 기록한다.

```markdown
| 회차 | 관계 | 변화 | 겉으로 보이는 행동 | 실제 감정/이해관계 | 다음 갈등 |
|---|---|---|---|---|---|
|  |  |  |  |  |  |
```

## 수정 우선순위

설정 충돌이 생기면 아래 순서로 고친다.

1. 독자 계약을 깨는 충돌.
2. 주인공 능력이나 한계가 뒤바뀌는 충돌.
3. 인물 동기가 갑자기 바뀌는 충돌.
4. 장기 떡밥 회수가 불가능해지는 충돌.
5. 이름, 날짜, 위치 같은 표면 정보 충돌.

작은 오류를 고치려고 장르 보상이나 회차 훅을 망치지 않는다. 먼저 독자가 느끼는 약속을 지킨다.
